### 环形地图形式

场景使用“环状地图”，即场景本身是一个长卷轴，但是在首尾处进行特殊处理，使用传送链接首尾，并通过添加首尾的镜像对象来让玩家在地图末尾能看到地图起点的动态对象。

### 随机地图采用“混合式生成”（TileMap + 预制组件 + 装饰）

- **TileMap（基础层）**：负责高频重复、性能敏感、需要规则化的内容。
	- 地表与基本地形起伏
	- 可站立平台与静态阻挡（核心碰撞）
	- 基础地形材质过渡（草/泥/岩等）
- **预制组件 Prefab Chunks（结构层）**：负责大结构与玩法关键点。
	- 建筑、掩体、防御工事、大型障碍物
	- 任务目标点（虫巢、工厂、据点等）
	- 需要独立碰撞/脚本/动画的对象
- **装饰层（点缀层）**：负责氛围与细节。
	- 小型装饰瓦片：弹坑、残骸、植被
	- 特效标记点：烟雾、火焰
	- 动态生成点：敌人、资源箱、交互点

### 结论：地图生成的基本单位与组合规则（替换讨论）

> **结论一句话**：把“可站立地形（平台层）”从纯 Tile 过渡为 **参数化的 Chunk（区块）拼接**；TileMap 继续负责装饰与少量规则化填充，但**不再承担跨地形边界的自动过渡裁决**。
> 

#### A. 为什么改成 Chunk 为主（你当前遇到的问题的直接解法）

- 不同地形交界处的自动地形过渡容易错位，根因是：边界处既需要拓扑一致（碰撞、可走线），又需要美术过渡一致（角、边、连接件）。把“边界”做成**可控的预制区块**更稳。
- Chunk 自带完整边缘与碰撞，拼接时只做**连接校验**，不会在交界处触发 Tile 逻辑错误。

#### B. 场景拆分：BattleMap（完整场景） vs MapParts（可拼合部分）

- **BattleScene.tscn（战斗场景，固定）**
	- 相机、UI、玩家出生与系统管理器
	- 只依赖 MapData（不关心地图是固定还是随机）
- **BattleMapRoot（地图根节点，运行时生成/挂载）**
	- **Sky/UltraFar/远景**：由 *MapTheme*（全局参数）决定并一次性创建
	- **拼合区域（中景/平台层/平台前遮盖等）**：由 Chunk 规则生成
	- 其余如装饰/刷怪点：属于生成数据的一部分

#### C. Chunk 的规格（最关键的“基本单位”定义）

1. **Chunk 长度（允许变长，但必须离散对齐）**
	- Chunk 可变长（例如 720px、1920px，未来也可更长）。
	- 但拼接点必须对齐到一个统一的 **Snap Grid**：`snap_step_px`。
	- 测试阶段建议：`snap_step_px = 240`（与你当前 480×270 的屏幕视觉比例匹配，且便于把一屏拆成 2 个拼接单元）。
	- 规则：`chunk_length_px` 必须是 `snap_step_px` 的整数倍。
	- 备注：tile 依然用于绘制与碰撞，但“生成与拼接”的基本尺度以 `snap_step_px` 为准。
2. **Chunk 内容范围（只放“拼合必需内容”）**
	- 必需：平台层（碰撞）与必要的中景背板（不碰撞）
	- 可选：结构节点（掩体、据点）与标记点（spawn/目标/资源）
	- 不建议：天空、超远景这类全局层（它们由 MapTheme 统一决定）
3. **高度档位（HeightTier）与过渡职责**
	- 高度档位固定为：`LOW / MID / HIGH`。
	- **MID** 作为默认基准高度（最常用平地），LOW 为洼地档位，HIGH 为高地档位。
	- **只有 Transition Chunk 允许改变高度档位**（负责上坡/下坡）。
4. **Chunk 的“连接口”概念（保证拼接不出错）**
	- 每个 Chunk 定义左右边界的连接口（Port）：
		- `left_port`：`surface_theme`（Rainforest/Base）、`height_tier`（LOW/MID/HIGH）、主通路约束（宽度等）
		- `right_port`：同上
	- 连接规则：`A.right_port` 必须能与 `B.left_port` 匹配（主题兼容 + 高度差规则 + 通路不断）

#### C.1 测试阶段（MVP）硬规则（强烈建议写死）

- Rainforest 的 **Filler** 允许使用 `LOW/MID/HIGH`（即使 HIGH 素材暂时缺失，也可以让 HIGH 档位存在于规则中，但不进入候选池）。
- **所有 POI 与 Objective 的端口高度固定为 `MID → MID`**（统一插入点）。
- 所有高度变化一律由 Transition Chunk 完成（包括“任意高度 → MID”与“MID → 任意高度”）。

#### D. 生成规则（可落地的最小版本）

> 尺寸表达建议：由于 Chunk 可变长，不建议再用 `chunks_x` 作为地图长度单位；改用 `map_length_steps`（总步数，步长 = `snap_step_px`）或 `map_length_px`（总像素长度）。
> 

> **测试阶段（MVP）简化约束**：只做 *Rainforest（主地形）* + *Base（任务地形）* 两类拼合地形；POI 只在雨林里出现；Objective 只在基地里出现；沙漠/殖民地先不做。
> 
1. **决定全局 MapTheme（一次性）**
	- `biome_id`（雨林/沙漠/虫巢）
	- `time_of_day`（白天/黄昏/夜）
	- `weather_id`（可选）
	- 由它决定：天空背景、远景素材集、色调（CanvasModulate）、雾化参数
2. **决定地图大小与环形结构（一次性）**
	- 地图是 X 方向环形：`chunks_x` 个 Chunk 首尾相接
	- Y 方向固定高度：`chunks_y` 先保持 1（平台主层），后续再扩展
	- `size_tier`（由难度或任务等级映射）决定 `chunks_x` 的范围
3. **按“主节奏”选取 Chunk 序列（核心）**
	- 定义一个抽象的节奏序列（Beat），例如：
		- `Travel`（行进）
		- `Encounter`（战斗）
		- `POI`（兴趣点）
		- `Objective`（任务点）
		- `Transition`（地形过渡段，专用 Chunk）
	- 生成器先生成 Beat 序列，再为每个 Beat 从库里挑选满足端口条件的 Chunk
4. **地形交接用“Transition Chunk”显式处理（你的痛点的核心解法）**
	- 例如：草地→岩地、沙漠→岩地，不要依赖 Tile 自动过渡
	- 每一对（或一组）地形组合都做少量 Transition Chunk：
		- 平地过渡
		- 上坡过渡
		- 下坡过渡
	- 这样拼接后平台碰撞与画面边缘都可控
5. **二次填充（可选，保持 Tile 的价值）**
	- 在 Chunk 拼好后，再由 TileMap 做“安全的填充”:
		- 不改变碰撞轮廓的地表细节
		- 小装饰与碎片
		- 轻量的材质噪声（只在同一地形内部，不跨边界）

#### E. 数据结构建议（用于复现与联机）

- **MapDefinition**（定义）
	- `generator_id`
	- `generator_version`
	- `world_seed`
	- `size_tier` / `chunks_x`
	- `theme`（biome/time/weather 等）
- **MapData**（结果）
	- `chunk_sequence[]`：每段使用的 chunk_id 与变体参数（flip、装饰密度等）
	- `poi_list[]` / `objective_list[]`
	- `spawners[]`

> 关键点：客户端只要有 MapDefinition 与同一份 Chunk 库，就能确定性生成同一份 MapData。
> 

#### F. 当前概念信息的缺口清单（后续我们要补齐的）

1. **地图分类（MapType）**
	- 例如：雨林地表型、沙漠地表型、虫巢洞穴型
	- 每类需要的 Chunk 族群与端口规则不同
2. **参数表（MapGenConfig）**
	- `size_tier` 的离散档位与 `chunks_x` 范围
	- Beat 权重与最小间距（两个 POI 之间至少隔几个 Travel）
	- 各类 Chunk 的库容量目标（MVP 先做多少）
3. **地形类型与兼容矩阵**
	- `TerrainType` 列表（草/沙/岩/洞穴…）
	- 哪些可以直接相接，哪些必须经过 Transition
4. **端口约束阈值**
	- 高度差阈值
	- 允许的坡度
	- 主通路最小宽度
5. **MVP Chunk 清单**（非常建议先锁定）
	- 每个地形：Travel 平地/上坡/下坡（各 1-2 个变体）
	- 每对常见交接：Transition（平地为主）
	- 1 个 POI（据点或虫巢）
	- 1 个 Objective（任务点）

### 交互物表现方式：瓦片 + 预制组件混合

采用该方案的原因：

- **开发效率高**：可先用瓦片搭原型，结构库后续逐步补齐。
- **表现力与可玩性平衡**：瓦片保证规则与性能，预制件补足艺术与变化。
- **可扩展**：新增结构/目标点只需扩充预制件库与放置规则。

不推荐的方向：

- **纯“大型地块/房间拼接”**作为主方案（成本高、依赖资源量，变化性取决于地块数量）。

### 生成流程按 3 步拆分，便于迭代与调参

1. **生成基础地形（TileMap）**
	- 使用 Noise / WFC / 规则系统均可。
	- 核心目标：得到“可走/不可走”的地形骨架，并保证连通性与基础节奏（开阔区、狭窄区、上坡下坡）。
2. **放置结构预制件（Prefab Chunks）**
	- 根据规则选择位置：
		- 不与出生点重叠
		- 不破坏主通路
		- 控制密度与最小间距
	- 对结构边缘做“融合”：使用过渡瓦片、遮罩、或结构自带边缘装饰。
3. **撒装饰与动态点位（Decor + Spawners）**
	- 用权重与约束控制装饰频率。
	- 将敌人/资源/特效点位视为生成数据的一部分（便于复现与调参）。

### 落地规范：每个“潜兵场景”（战斗或大厅）的分层要求

> 目标：统一美术与工程的层级口径，保证视差、遮挡与可读性。
> 

#### 必备层（推荐以 Parallax/分层 TileMap 组织）

- **天空层**（非 TileMap 亦可）
	- 建议：不动或极小移动。
- **超远景层**（TileMap，视差慢）
	- 用途：地平线级景物。
- **远景层**（TileMap，视差较慢）
	- 用途：平台之外的氛围景物，避免与可站立轮廓冲突。
- **中景层**（TileMap，同速）
	- 用途：平台后方延伸与结构背板。
	- 规则：不承载碰撞，不做“看起来能站”的高对比边缘。
- **平台层**（TileMap，同速，核心碰撞）
	- 用途：可站立平台与主要阻挡物。
- **平台前部遮盖层**（TileMap，同速）
	- 用途：向镜头（画面底端）延伸的透视/断面。
	- 规则：避免大面积遮住角色主体，尽量控制在脚部附近或镜头底部区域。
- **前景层**（可 TileMap 或场景节点）
	- 用途：可遮挡镜头的装饰物，增强景深与氛围。
	- 规则：战斗空间减少高对比遮挡，避免遮住关键信息。

#### 平台脚下装饰（两层，用于“草/碎石”表演）

- **脚下远层装饰**（角色上方遮它之前，先被角色遮）
	- 用途：远处草、碎石，增加地表层次。
- **脚下近层装饰**（在角色之上，能遮住角色）
	- 用途：近处草、碎石，营造“角色在环境里”的感觉。
	- 规则：密度要低，避免持续遮挡角色轮廓。

#### 推荐的视差倍率（作为统一基准，可后续微调）

- 天空：0.00～0.05
- 超远景：0.10～0.20
- 远景：0.25～0.40
- 中景（平台后方延伸）：1.00
- 平台层：1.00
- 平台前部遮盖：1.00
- 脚下装饰两层：1.00
- 前景：1.00（或 1.10～1.25 形成“贴镜头”效果）

	- 草丛、碎石、残骸、弹坑

### 工程建议（约束与分工）

- **TileMap 负责“静态可碰撞”与“高重复可渲染”内容。**
- **需要脚本/动画/独立状态的对象用场景节点（预制件）放置。**
- **生成算法只决定“地形类型分布 + 结构落点 + 装饰密度”，视觉过渡尽量交给地形/规则系统处理。**

### 地图类型与同步方案讨论（固定地图 vs 随机地图）

#### 1) 两种地图在逻辑上的“共同抽象”：MapDefinition / MapData

无论是固定地图还是随机地图，建议都落到同一种运行时数据结构里：

- **MapDefinition（定义）**：本局地图“是什么”。
	- 固定地图：直接引用一个预设资源（场景/TileMap资源/关卡配置）。
	- 随机地图：引用“生成器 + 参数 + 种子”。
- **MapData（结果）**：本局地图“生成/加载后的结果”。
	- 例如：每个 TileMap 图层的用到的瓦片ID分布、放置的Prefab列表、出生点/目标点/刷怪点列表等。

这样你的“战斗场景”可以只依赖 MapData 来摆放与初始化内容，避免关心它来自固定还是随机。

#### 2) 场景组织：同一个 BattleScene，地图只是它的子节点

你的直觉是对的：随机地图“最开始不存在”。

推荐结构：

- **BattleScene.tscn**：只包含相机、玩家出生、UI、系统管理器等通用结构。
- **MapRoot（Node2D）**：开局时被填充。
	- 固定地图：实例化 Map_XXX.tscn 作为子节点挂上来。
	- 随机地图：运行生成器，创建/填充 TileMap、实例化 prefab chunks、摆 spawner。

这样“同一个场景”可以承载两种地图模式。

#### 3) 随机地图同步：优先用 Seed + 版本化参数（每个客户端本地生成）

多人同步时，最稳的默认策略是：

- **主机（Host）只负责决定**：
	- 关卡生成器ID（generator_id）
	- 种子（seed）
	- 生成参数（参数集/难度/biome等）
	- 生成器版本（generator_version 或 content_hash）
- **每个客户端收到这些信息后本地生成同一份 MapData**。

前提条件：

- 生成算法必须**确定性（deterministic）**。
- 生成依赖的数据（瓦片集、prefab库、权重表）在所有客户端必须一致。

> 实践建议：把参数集做成一个可序列化的“关卡配置（MapGenConfig）”，再加上 generator_version 做校验，避免后续改规则导致旧seed复现不出来。
> 

#### 4) 什么时候不适合只同步 Seed？（需要回退到“同步结果”）

以下情况会导致“同seed但结果不一致”：

- 使用了不确定性的随机源（例如混入时间、帧序、节点遍历顺序）。
- 使用浮点运算且不同平台可能存在差异（一般2D瓦片生成问题不大，但要注意）。
- 生成过程依赖资源加载顺序、字典遍历顺序等。

解决办法通常有两种：

1. **固定随机源与顺序**：所有随机都来自同一个 PRNG，所有列表在使用前排序，避免遍历顺序差异。
2. **主机下发“生成结果摘要/差量”**：例如下发 prefab 放置列表与关键点位，TileMap 只下发关键区域或压缩数据。

#### 5) 推荐的网络分工（你现在的项目阶段的最小实现）

> 目标：先做一个**稳定可联机**的版本，减少“确定性生成 + 复杂回滚”的坑。
> 

**阶段 A（MVP，推荐先落地这个）**

- **Host 决定地图**
	- 固定地图：`fixed_map_id`
	- 随机地图：`generator_id + seed + MapGenConfig + generator_version`
- **客户端加入时**
	- 收到上述 MapDefinition
	- 本地生成地图（或加载固定地图）
	- 进入 Ready（只表示“地图就绪 + 资源就绪”）
- **战斗开始**
	- 所有人 Ready 后，Host 广播 Start

**阶段 B（地图稳定后再做的增强）**

- **生成结果校验（hash）**
	- 客户端生成完计算一个 hash（对关键点位与 prefab 列表做 hash 即可）
	- 回传给 Host 做一致性检查
- **必要时回退到“下发结果”**
	- 如果某些设备/版本导致不一致，Host 下发 prefab 放置列表与关键点位作为修正

#### 6) 动态实体（敌人/掉落/可破坏物）的权威与同步（明确方案）

> 结论：采用 **Host 权威（Host-authoritative）**。客户端会创建实体用于渲染与碰撞，但不裁决结果。
> 

**6.1 敌人生成（Spawn）规则**

- **生成权威在 Host**
	- Host 决定“何时在何处刷出什么敌人”，并分配唯一 `enemy_id`。
	- Host 广播 `SpawnEnemy(enemy_id, type_id, position, initial_state, optional_seed)`。
- **客户端的行为**
	- 收到 Spawn 后创建本地敌人节点（非权威副本）。
	- 其生命值、状态机关键状态以 Host 同步为准。

**6.2 敌人状态同步（State Replication）最小集合**

- Host → Clients（定频或事件驱动）同步：
	- `enemy_id`
	- `position / velocity`（或 transform）
	- `facing / aim_dir`（若影响表现）
	- `hp`（或血量段/是否濒死）
	- `state`（Idle/Move/Attack/Stun/Dead 等）
- 客户端做：
	- 插值/平滑（避免抖动）
	- 动画播放与特效（可预测，但最终以 Host 为准）

**6.3 伤害与命中（Damage / Hit）推荐流程（MVP 版）**

- 客户端上报意图：`HitRequest(attacker_id, enemy_id, attack_id, client_time, optional_hit_pos)`
- Host 做裁决：
	- 校验合法性（距离/冷却/是否可见等）
	- 计算伤害，更新 HP，决定死亡
	- 广播 `DamageResult(enemy_id, new_hp, flags)` 与 `Death(enemy_id)`

> 备注：如果你短期内不做高强度 PvP，只做合作 PvE，这套“Host 裁决 + 客户端表现”足够好用。
> 

**6.4 掉落物与可破坏物（建议同样 Host 权威）**

- 掉落：由 Host 生成并分配 `drop_id`，客户端只显示与拾取请求。
- 可破坏物：
	- 若是少量关键物件，用节点实体同步（类似敌人）。
	- 若是大面积地形破坏（你当前并不打算做），成本会指数上升，建议避免。

**6.5 与 Seed 的关系怎么用（不和“地图静态”混在一起）**

- Seed 负责：
	- 生成刷怪点位/据点配置（SpawnPoints / OutpostConfig）
- Host 仍负责：
	- 具体刷怪时机与敌人实例的 `enemy_id`
	- 以及最终伤害结算

	- 或者demo阶段可以只提供训练场固定地图，和固定的难度的随机地图（没法选择，但是可以随机生成种子），这样我们可以准备一个雨林，一个沙漠场景地图。

	- 不需要录像

#### 8) 地图难度 × 尺寸（与种子、区块的关系）建议（简短版）

> 目标：让“难度选择”不仅影响敌人强度，也能稳定地映射到**地图的总尺寸/区块数量**，并且保持可复现。
> 

**8.1 用“尺寸等级（size_tier）+ 区块网格”来描述地图大小**

- 先定义 TileMap 的**最小区块**（Chunk），例如：
	- `chunk_w_tiles × chunk_h_tiles`（比如 64×32 或 64×64，按你的平台游戏节奏定）
- 再用区块数量表达总尺寸：
	- `chunks_x × chunks_y`
- 让“难度/任务等级”映射到一个离散的 `size_tier`，由它决定区块数量范围。

**8.2 推荐的映射方式：难度决定 size_tier，seed 决定具体布局**

- `difficulty_id`（或 mission_level） → 选择 `size_tier` 与参数范围
- `seed` → 在该参数范围内抽取具体 `chunks_x/chunks_y`，并生成每个 chunk 的内容

这样可以做到：

- 同一难度下地图大小“有变化但可控”
- 同一个 (difficulty_id, seed) 必然复现同一张图

**8.3 seed 的拆分：全局种子 + 区块子种子（避免改局部影响全局）**

- 使用一个 `world_seed`（Host 下发）
- 对每个区块派生 `chunk_seed`：
	- `chunk_seed = hash(world_seed, chunk_x, chunk_y, generator_version)`
- 生成时尽量让每个区块的随机只依赖自己的 `chunk_seed`

收益：

- 以后你想“加一层装饰算法”或“替换某个区块规则”，不至于把整张图的随机序列扰乱得完全不同。

**8.4 难度除了尺寸，还应该影响什么（保持设计弹性）**

- 结构密度（掩体/建筑/障碍物密度）
- 关键点数量（据点/目标点/资源点数量）
- 路径连通性的约束强度（主通路更宽或更窄）

> 备注：当前是“循环地图”，可以把循环方向设为 X 方向环形（chunks_x 环），Y 方向固定高度（chunks_y 小），更符合平台玩法。
>
