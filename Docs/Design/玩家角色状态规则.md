# Player State Rules

> 本文定义玩家角色的状态层、状态切换、优先级、打断规则、受击规则、死亡规则，以及与输入/表现层的边界。
> 
> 
> 本文是玩法规则文档，不是代码实现文档。
> 
> 输入动作语义见 `player_input_guide.md`
> 
> 角色职责与生命周期概览见 `player_character.md`
> 
> 表现与动画策略见 `player_visual_animation.md`
> 

---

# 1. Design Goal

玩家状态系统需要满足以下目标：

- 让输入语义清晰可控
- 支持多层状态组合，而不是一个巨大状态枚举
- 支持受击、打断、死亡、攀爬、飞扑等复杂动作冲突
- 支持未来扩展武器、异常状态、联机同步
- 让动画系统服务于状态，而不是替代状态

---

# 2. Core State Model

玩家状态采用 **分层状态模型（Layered State Model）**。

当前至少包含以下四层：

- `LifeState`
- `ActionState`
- `PostureState`
- `AimState`

这些状态层可以组合存在。

例如：

- `LifeState = Alive`
- `ActionState = Reloading`
- `PostureState = Crouch`
- `AimState = NotAiming`

---

# 3. State Layers

## 3.1 LifeState

描述角色是否仍然作为可控战斗单位存在。

Possible values:

- `SpawnIn`
- `Alive`
- `Dead`
- `WaitingReinforce`

### SpawnIn

角色正在进入战场。

特点：

- 输入通常锁定
- 角色不应执行普通战斗动作
- 可播放出生/重生演出
- 演出结束后进入 `Alive`

### Alive

角色正常参与战斗。

### Dead

角色已死亡。

特点：

- 普通输入失效
- 普通动作结束
- 不再参与正常战斗控制
- 进入死亡表现与尸体替换流程

### WaitingReinforce

角色等待队友复活 / 支援重生。

特点：

- 不参与战斗
- 不接受普通输入
- 等待重新进入 `SpawnIn`

---

## 3.2 ActionState

描述角色当前执行的主要动作状态。

Possible values (current design):

- `Idle`
- `Moving`
- `PostureChanging`
- `Reloading`
- `WeaponSwitching`
- `Diving`
- `Climbing`
- `Dropping`
- `HitReact`
- `HeavyHit`

Future possible values:

- `Interacting`
- `UsingStratagem`
- `MeleeAttack`
- `Pinned`
- `Knockdown`

### Notes

- `ActionState` 主要决定控制权归属
- 同一时间通常只有一个主要 `ActionState`
- 某些轻微效果不一定要升格为 `ActionState`，可以作为 Effect 存在

---

## 3.3 PostureState

描述角色身体姿态。

Possible values:

- `Stand`
- `Crouch`
- `Prone`

规则：

- 姿态切换是离散的，不是连续量
- 基本链路为：

`Stand -> Crouch -> Prone`

`Prone -> Crouch -> Stand`

### Posture semantics

### Stand

- 标准移动姿态
- 轮廓最大
- 通常移动能力最好

### Crouch

- 中间姿态
- 兼顾机动与掩体利用

### Prone

- 最低姿态
- 轮廓最小
- 机动能力最弱
- 某些动作受限

---

## 3.4 AimState

描述角色是否处于瞄准模式。

Possible values:

- `NotAiming`
- `Aiming`

### NotAiming

- 鼠标位置不影响角色朝向
- 当 `move_input != 0` 时，朝向由移动输入决定

### Aiming

- 角色朝向由鼠标左右位置决定
- A / D 不再改变朝向
- A / D 变为相对朝向的前后移动

---

# 4. Core Rules

## 4.1 Facing Rules

角色朝向 `Facing` 只有两个合法值：

- `Left`
- `Right`

### When AimState = NotAiming

- 若 `move_input < 0`，则 `Facing = Left`
- 若 `move_input > 0`，则 `Facing = Right`
- 鼠标位置不影响朝向

### When AimState = Aiming

- 鼠标位于角色左侧时，`Facing = Left`
- 鼠标位于角色右侧时，`Facing = Right`
- A / D 不改变朝向

---

## 4.2 Movement Semantics

### NotAiming

- A = 左移
- D = 右移
- 输入同时控制移动和朝向

### Aiming

- A = 后退（相对当前朝向）
- D = 前进（相对当前朝向）
- 朝向保持由鼠标控制

### Prone + Aiming

- 当前设计：匍匐瞄准时禁用水平移动

## 4.3 Posture Change Rules

`posture_down`：

- `Stand -> Crouch`
- `Crouch -> Prone`
- `Prone -> Prone`

`posture_up`：

- `Prone -> Crouch`
- `Crouch -> Stand`
- `Stand -> Stand`

### Restrictions

- 某些 `ActionState` 下禁止姿态切换
- 一般来说：
	- `Climbing`
	- `Dropping`
	- `Diving`
	- `Dead`
	不允许进行姿态切换

---

## 4.4 Aim Entry / Exit Rules

### Enter Aim

角色尝试进入 `Aiming` 时，需要满足：

- `LifeState = Alive`
- 当前 `ActionState` 允许瞄准
- 当前武器允许进入瞄准模式

### Exit Aim

以下情况应强制退出 `Aiming`：

- 进入 `Reloading`
- 进入 `WeaponSwitching`
- 进入 `Climbing`
- 进入 `Dropping`
- 进入 `Dead`

# 5. State Priority

高优先级状态会压制低优先级状态。

Current priority recommendation:

1. `Dead`
2. `SpawnIn`
3. `Climbing / Dropping`
4. `HeavyHit`
5. `Diving`
6. `HitReact`
7. `Reloading`
8. `WeaponSwitching`
9. `Aiming`
10. `PostureChanging`
11. `Moving`
12. `Idle`

### Notes

- `LifeState` 优先级高于普通 `ActionState`
- 一旦进入 `Dead`，普通动作立即失效
- `Climbing / Dropping` 当前建议为强执行动作
- `Diving` 具有高优先级与高抗打断，用于保障战术闪避手感

# 6. Event-Driven Transition Model

状态切换不是由动画直接决定，而是由**事件触发 + 规则判断**决定。

基本流程：

```
Event Occurs
↓
State System Evaluates Conditions
↓
Transition Allowed?
↓
Change State
↓
Trigger Logic and Presentation
```

### 7. Event Categories

#### 7.1 Input Events

由输入触发：

- AimPressed
- AimReleased
- FireRequested
- ReloadRequested
- DiveRequested
- ContextActionRequested
- SwitchWeaponRequested
- InteractRequested
- PostureUpRequested
- PostureDownRequested

#### 7.2 External Events

由战斗或环境触发：

- HitReceived
- ExplosionReceived
- StatusEffectApplied
- DeathTriggered
- SpawnRequested

### 8. Interrupt System

本项目采用 Interrupt Power vs Interrupt Resistance 模型处理动作打断。

#### 8.1 Terminology

- Interrupt Power
	- 事件的打断强度。
- Interrupt Resistance
	- 当前状态的抗打断能力。
- Interruptible
	- 当前状态是否允许参与打断判定。

#### 8.2 General Rule

当角色处于可打断状态时：

- 若 InterruptPower > InterruptResistance
	- 则当前状态被打断
	- 进入新的反应状态或目标状态
- 若打断值不足：
	- 当前状态继续执行
	- 仍可附加轻微反馈效果

#### 8.3 Hard Rules vs Numeric Interrupts

不是所有状态都靠数值打断。

- Hard-rule states
	- 以下状态建议优先用硬规则处理：
		- Dead
		- WaitingReinforce
		- SpawnIn
		- Climbing
		- Dropping
- Numeric interrupt states
	- 以下状态适合进入打断值比较：
		- Idle
		- Moving
		- PostureChanging
		- Reloading
		- WeaponSwitching
		- Diving
		- Aiming（注意：通常不是主 ActionState，但可被强制退出）

#### 8.4 Recommended Initial Resistance Values

以下只是设计参考，不代表最终数值。

- Idle: 10
- Moving: 30
- PostureChanging: 25
- Reloading: 40
- WeaponSwitching: 40
- Diving: 80
- Climbing: non-interruptible
- Dropping: non-interruptible
- TODO-TUNE:
	- 这些值需要在实机测试后调整。

#### 8.5 Recommended Initial Interrupt Powers

- LightHit: 20
- MediumHit: 40
- HeavyHit: 60
- ExplosionShock: 80+
- TODO-TUNE:
	- 后续需要按敌人类型、武器类型、爆炸类型细分。

### 9. Hit Processing

受击应被视为事件，而不是天然状态。

也就是说：

- HitReceived 是事件
- HitReact / HeavyHit / Dead 是可能结果状态

#### 9.1 Hit Event Suggested Fields

建议 HitReceived 至少包含以下数据：

- damage
- interrupt_power
- hit_type
	- light
	- medium
	- heavy
	- explosive
- source_direction
- knockback_vector
- status_effects
- can_cause_death

#### 9.2 Hit Resolution Flow

当 HitReceived 发生时：

- 计算伤害
- 扣除 HP
- 检查是否死亡
- 若未死亡，则比较打断值
- 根据结果附加效果或切换状态

伪流程：

HitReceived
↓
ApplyDamage
↓
If HP <= 0
→ Dead
Else
Compare InterruptPower vs InterruptResistance
If insufficient
→ minor effect only
If sufficient
→ HitReact or HeavyHit

### 10. Minor Effects vs Major Reaction States

受击不一定总要切换 ActionState。

#### 10.1 Minor Effects

这些可以只作为短时效果，不一定进入完整动作状态：

- 准星扰动
- 短暂精度下降
- 轻微闪白
- 轻微摄像机抖动
- 小幅朝向扰动

Aim Disturbance

AimDisturbance 建议作为**短时效果（Effect）**处理，而不是主 ActionState。

它用于模拟：

- 轻武器命中后的瞄准晃动
- 准星不稳定
- 短时间的瞄准干扰

### 11. Reaction States

#### 11.1 HitReact

中等受击反应状态。

特点：

- 短暂打断当前动作
- 可短时间限制输入
- 可播放受击动画
- 可附带轻微位移 / 后退
- 状态结束后回到正常控制

适合用于：

- 普通中等火力命中
- 打断换弹
- 打断切枪
- 打断普通移动

#### 11.2 HeavyHit

重受击状态。

特点：

- 强制终止当前动作
- 可施加击退或击飞
- 可禁用输入更长时间
- 可播放更明显的受击动画

适合用于：

- 爆炸
- 重型近战命中
- 高冲击武器命中

### 13. Climb / Drop Rules

#### 13.1 Entry Rules

Climbing / Dropping 由 ContextActionRequested 触发，但必须满足环境条件。

#### 13.2 Forced Stand Rule

若角色当前姿态为：

- Crouch
- Prone

则在进入 Climbing / Dropping 之前：

- 先强制回到 Stand
- 再执行攀爬 / 下落动作

#### 13.3 Restrictions

在 Climbing / Dropping 期间：

- 禁止普通移动
- 禁止姿态切换
- 禁止瞄准
- 禁止开火
- 禁止换弹
- 禁止切枪

#### 13.4 Interruptibility

当前建议：

- Climbing / Dropping 先作为不可打断状态处理

原因：

- 动作依赖受控动画和落点
- 中途打断容易造成位置错误和表现异常

#### 14.2 Forced Exit from Aim

进入 Reloading 时：

- AimState 必须退出 Aiming

#### 14.3 Fire Restriction

Reloading 期间禁止开火。

#### 14.4 Movement Restriction

### 16. Death Rules

#### 16.1 Death Condition

当 Current HP <= 0 时：

- LifeState = Dead
- 该状态具有最高优先级。

#### 16.2 On Enter Dead

进入 Dead 后：

- 终止所有普通动作
- 终止普通输入
- 退出瞄准
- 停止换弹
- 停止切枪
- 不再允许姿态切换
- 不再作为正常战斗单位行动

#### 16.3 Death Presentation Type

死亡后的表现由死亡类型决定。

建议至少区分：

- StandardDeath
- ExplosiveDeath

StandardDeath

- 播放普通死亡反馈或短倒地起势
- 生成尸体对象

ExplosiveDeath

- 直接生成爆散/碎裂组件
- 不要求播放标准倒地动作

### 17. Corpse Replacement Rules

活体角色与尸体表现对象必须分离。

推荐流程：

↓
Spawn PlayerCorpse or CorpseFragments
↓

↓

尸体对象只负责：

- 表演
- 物理效果
- 碎裂组件

不再负责角色控制与战斗逻辑。

### 20. Presentation Boundary

状态系统负责：

- 状态定义
- 转移条件
- 输入可用性
- 动作可用性
- 打断判定
- 生命周期推进

动画 / 表现系统负责：

- 播放与过渡
- 关键帧事件
- 外观反馈
- 尸体或碎裂表演

原则：

打断本质上是逻辑状态切换。
动画、尸体替换、击飞、爆炸碎裂等都属于状态切换后的表现与执行结果。

- Spawn / Respawn
	- 出生无敌时间：下落和爬出动画应该给予无敌时间
	- 出生演出期间是否允许微调输入：出生时，玩家会经历多个阶段，从空中下落--空头仓落地，玩家角色正式从空头仓爬出，成为可控制的角色
- Death
	- 标准死亡时是立即替换尸体，还是先给极短死亡起势：立即替换即可，但是尸体会被增加一定的动画表演，玩家可以看到这些死亡动画（重生延迟期间），但不因为尸体的表演会影响其重生和部署的速度。
	- 爆炸死亡的碎裂程度与组件数量：人物和敌人均最多只有个位数的组件，玩家和敌人的可动组件会多一些，但是用于死亡表演的组件会简化，比如手臂会被视为一个整体，而不区分小臂和手掌等。
