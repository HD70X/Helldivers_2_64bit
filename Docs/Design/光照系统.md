### 1. 目标（Goals）

- 支持在 2D 平台射击战斗中快速读图：角色与关键交互物清晰可见。
- 通过少量关键光源营造氛围，但不牺牲可玩性与性能。
- 允许不同关卡选择不同光照风格（无固定光照 / 静态装饰光 / 太阳光氛围）。

### 2. 非目标（Non-goals）

- 不追求全局一致的“真实感”光照。
- 不做像素级可破坏地形那种对光照/遮挡的极端精度要求。
- 当前阶段不做完整的静态光照烘焙流程（仅预留扩展位）。

### 3. 当前设计结论（现阶段采取的方案）

#### 3.1 战斗场景（关卡级光照）

- **默认：不设置固定的场景光照**。
	- 目的：避免“关卡必须按固定灯光美术”绑定，先把战斗可读性与核心玩法跑通。
- **预留：静态装饰光（例如路灯）**
	- 仅在具体场景需要时启用。
	- 倾向：用于引导路线、强调地标、增强氛围。
- **预留：模拟太阳光 / 大范围氛围光**
	- 用于让装饰物投下阴影或形成明暗对比。
	- 是否启用取决于关卡主题与可读性需求。

#### 3.2 角色（随身基础光源）

- 角色自带 **圆形点光源（PointLight2D）**，用于略微照亮周围。
- 该基础光源 **不产生阴影**。
	- 理由：保持性能稳定，并避免角色周围频繁阴影跳动影响读图。

#### 3.3 武器与挂载（动态光源）

当角色持有武器（实例化子场景）或挂载组件时：

- **枪口闪光（Muzzle Flash Light）**
	- 由点光源 + 能量/亮度的短时变化实现。
	- **动态光源**，**有阴影**。
- **手电筒（Flashlight）**（若挂载）
	- **动态光源**，**有阴影**。
- **激光发射器（Laser Emitter）**（若挂载）
	- **动态光源**，**无阴影**。
	- 备注：激光本身更像“特效/线段 + 命中反馈”，阴影收益低，先不做以换取性能与实现简化。

### 4. 光照规则与约束（Rules & Constraints）

#### 4.1 目标平台与帧率目标

- 目标平台：**PC**。
- 推荐帧率目标：
	- **优先目标：60 FPS 稳定**（更适合动作射击的操作手感，也便于后续移植）。
	- **可选增强：120 FPS**（只作为“高配可达成”目标，不作为硬性设计约束）。
	- 设计约束默认按“60 FPS 稳定”来做取舍。

#### 4.2 同屏光源预算（建议的上限与分级）

考虑最多 **4 名玩家同屏**，以及场景装饰光：

- **动态光源（玩家相关）**
	- 每名玩家：
		- 角色基础光：1（无阴影）
		- 手电筒：0 或 1（有阴影）
		- 枪口闪光：0 或 1（有阴影，短时）
		- 激光发射器：0 或 1（无阴影）
- **静态装饰光（路灯等）**：每屏约 **2–4 个**（是否带阴影取决于成本，默认不带阴影或用更轻量设置）。
- **环境主光（太阳/月光）**：最多 **1 个**。

> 重要原则：**“带阴影的动态光”是最贵的资源**。同屏数量必须受控。
> 

#### 4.3 阴影开关规则（强制约束）

- 角色基础光：**不产生阴影**（固定）。
- 激光发射器：**不产生阴影**（固定）。
- 枪口闪光：
	- **默认：不产生“场景阴影”**（不与 TileMap/装饰物 occluder 交互）。
	- 可选增强（有性能余量才做）：只做 **角色局部阴影/角色自阴影** 的效果，不参与场景遮挡。
	- 降级策略：若已开启任何形式的阴影，仍允许在压力场景下强制关闭。
- 环境主光（太阳/月光）：
	- 默认：允许产生阴影，但必须遵循“选择性遮挡物参与”（见 6.2）。
	- 最坏情况：允许 **关闭阴影**，只保留氛围明暗。
- 手电筒：
	- 默认：**产生阴影**。
	- 如果出现性能风险：优先保留手电筒阴影，其他光源先降级。

#### 4.4 阴影优先级（从高到低）

1. 手电筒阴影（玩法读图价值最高，且是玩家可控的“关键时刻能力”）
2. 环境主光阴影（氛围价值高，但允许在最坏情况下关闭或做选择性交互）
3. 枪口闪光阴影（最不重要，优先降级，甚至默认不做场景阴影）
4. 静态装饰光阴影（通常优先关闭或降低质量）
5. 角色基础光 / 激光（本阶段禁用阴影）

#### 4.5 光照距离与范围（以 1920×1080 的屏幕尺度描述）

- 开发基准分辨率：**1920×1080**。
- 手电筒：最远约 **半个屏幕宽**（强读图/探索）。
- 枪口闪光：主要影响 **玩家周围**，范围约 **2 个玩家宽**（短时，不做大范围照明）。
- 激光：距离可以很远，但主要表现是“条状/线状特效 + 命中反馈”，不依赖阴影。

### 5. 体验与可读性（Gameplay Readability）

- 优先保证：
	- 玩家与敌人的轮廓在主要战斗区可被识别。
	- 关键地形边缘（落脚点、坑、台阶）不被阴影完全吞没。
- 手电筒是“玩法工具”，不只是装饰：
	- 光束范围内应尽量保持清晰。
	- 阴影可以增强深度感，但不能遮挡敌我识别。
- 枪口闪光是瞬态信息：
	- 用于提示开火与强化节奏。
	- 不应长时间照亮大范围，避免影响敌人轮廓判断。

### 6. 性能与优化策略（Performance）

#### 6.1 关于“实时投影 vs 手绘阴影”的取舍建议

你提出的思路（用方向光/点光让装饰物实时投影，避免手绘阴影）是合理的，但需要一个明确的兜底：

- **优先方案（推荐）**：环境主光“能投影，但成本受控”。
	- 让少量关键装饰物投影即可。
	- 避免让碎石这种高密度小装饰都投影。
- **兜底方案**：当同屏 occluder 过密或帧率波动明显时：
	- 环境主光 **关闭阴影**，只保留明暗氛围（仍可保留方向性光感）。
	- 或改用 **手绘/贴图阴影**（装饰物自带暗部），实时光只做加成。

#### 6.2 装饰物分级与 Occluder 规范（美术/关卡可执行）

目标：用最低的制作与性能成本，得到“看起来有层次”的阴影氛围。

#### A 级：大型遮挡物（必须参与阴影交互）

**定义**：轮廓明显、体积大、会形成清晰投影的装饰物。

- 示例：大石块、墙体突出结构、巨大植被、路牌/大机器、拱门、粗大立柱。
- 规则：
	- **建议设置 occluder**，允许参与 **环境主光** 与 **手电筒** 的阴影。
	- occluder 形状要“少边、贴轮廓的大形”，避免细小凹凸。
	- 优先保证“阴影边缘稳定”，而不是完全贴图形状。

#### B 级：中型地标装饰（选择性参与）

**定义**：有辨识度，但尺寸或数量中等，投影存在感适中。

- 示例：路灯、广告牌、路边栏杆、较大的碎石堆、稀疏灌木。
- 规则：
	- 默认 **不做 occluder**。
	- 只有在“作为地标/构图中心”的少数情况下，才给 occluder。
	- 若开启 occluder：只让它参与 **环境主光**（可选），并尽量不影响战斗核心区域的读图。

#### C 级：小型高密度碎物（禁止参与阴影交互）

**定义**：数量多、体积小、分布密或随机的小装饰。

- 示例：碎石、草屑、小花、小枝条、沙漠小植被点缀。
- 规则：
	- **禁止设置 occluder**（默认）。
	- 阴影表现方式采用：
		- 贴图自带暗部（手绘阴影/暗边）。
		- 或在 sprite 内做简单明暗（色块/渐变），不走实时阴影。
	- 理由：收益低、噪声高、性能与调试成本远大于细节价值。

#### D 级：纯氛围贴片（永不参与）

**定义**：远景、背景层、粒子/贴花类，仅用于氛围。

- 示例：背景山体剪影、远处云层、灰尘粒子、地面 decal。
- 规则：
	- 不参与光照遮挡，只吃简单的整体明暗（例如 CanvasModulate）。

#### 关卡制作的快速检查（Checklist）

- 屏内 occluder 是否主要来自 **A 级**？
- 是否出现大量 C 级物件误加 occluder？
- 如果环境主光阴影开启，画面是否因碎物导致“阴影噪声”？
	- 如果是：优先删掉碎物 occluder，而不是调阴影质量。

#### 建议的“最省事默认值”

- 白天关卡：
	- 环境主光可以只做氛围明暗，阴影可关或只让 A 级参与。
- 夜晚/洞穴关卡：
	- 手电筒阴影优先保留。
	- 环境主光若造成波动，优先关闭阴影。

#### 6.3 动态降级（建议后续实现，但先写进设计）

- 当同屏“带阴影动态光”超过阈值时，按优先级自动降级：
	- 先关枪口闪光阴影。
	- 再降低环境主光阴影质量或范围。
	- 最后才考虑手电筒降级。

### 7. 风险与待定项（Open Questions）

- **环境主光实现形式**：方向光 vs 大范围点光？两者对氛围与性能的影响需要做一次小样机对比。
- **装饰物分级是否足够清晰**：是否需要把 A/B/C 级示例再补充到你当前的常用装饰物素材库？
- **装饰光（路灯）是否带阴影**：建议默认不带阴影，只有极少数“地标路灯”例外（可归类为 B 级中的特殊个例）。
- **多玩家同屏的最坏情况**：
	- 4 个玩家同时开手电 + 高频开火 + 屏内 2–4 路灯 + 1 环境主光。
	- 需要明确降级规则，确保 60 FPS 稳定。
- **激光的表现增强**：是否用“命中点亮斑/短时照亮”替代阴影来增加质感？
- **Occluder 密度的硬限制**：是否需要为关卡制作定一个“每屏 occluder 数量/复杂度”的美术规范？

## 实现参考（Implementation Notes）

> 这一节是“怎么做”的备忘，可随实现迭代而修改。
> 

### A. 基础配置（CanvasModulate + Light2D）

1. 场景中使用 `CanvasModulate` 作为全局环境光。
2. 使用 `PointLight2D` 作为角色光源或动态光源。
3. TileMap 通过 TileSet 的 Occluder 定义遮挡体（用于阴影）。

### B. 跟随玩家的基础光源（示例）

```
# 将 PointLight2D 作为玩家节点的子节点
extends CharacterBody2D

@onready var player_light: PointLight2D = $PointLight2D

func _ready() -> void:
	player_light.enabled = true
	player_light.energy = 1.5
	player_light.texture_scale = 1.0
	# 设计结论：基础光源不产生阴影
	player_light.shadow_enabled = false
```

### C. 常见问题排查（Checklist）

- 光照不显示：检查 `CanvasModulate` 是否太亮，或光源 `Enabled`。
- 阴影不工作：确认 TileSet 配置了 Occluder，多边形方向/层级正确。
- 性能问题：减少有阴影的光源数量，降低范围/能量，简化 Occluder。
