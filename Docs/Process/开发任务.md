# 开发任务

> 用于推进当前短期 Demo 的具体执行任务清单。
>
> 本文件聚焦“把现有原型包装成可试玩、可展示的 Early Demo”，
> 不在此阶段展开完整架构重构、联机落地或大规模系统扩展。

## 目标

在现有最小入口闭环已经建立的基础上，继续推进一个：

- 可进入
- 可理解
- 可试玩
- 可结束
- 可用于对外展示

的短期 Demo。

当前已知闭环基础：

- 游戏启动进入主菜单
- 主菜单可进入 `TestMap`
- `TestMap` 可返回主菜单

下一阶段重点不是新增大量系统，而是把现有内容组织成一条清晰、可验证、可扩展的试玩链路。

当前建议采用的 Demo 主线为：

1. 启动游戏
2. 进入主菜单
3. 进入 `TestMap`
4. 在场景中完成基础教学
5. 击毁一个最小训练目标
6. 显示完成结果
7. 返回主菜单或重新开始

这条主线成立后，项目就不再只是“测试场”，而是一个具备基础展示价值的 Demo。

---

## 一、当前阶段策略

### 1. 采用纵切优先，而不是功能散点补洞

当前阶段不建议把“HUD、角色、敌人、反馈”拆成完全独立的孤立工作项分别推进。

更合理的方式是围绕一条最小可试玩链路，做一条完整纵切：

- 教学提示
- 流程推进
- 基础战斗验证
- 完成判定
- 返回路径

必须能在同一轮试玩中被连起来验证。

### 2. 不为了短期快而制造后续高修改成本

当前可以接受“功能轻量”，但不应接受“结构随意”。

因此本阶段遵循以下约束：

- 不把 Demo 流程逻辑直接写死在 `character.gd`
- 不把 UI 更新逻辑硬塞进角色脚本或武器脚本
- 不为了一个教学场就做过重的通用任务系统
- 允许先做最小可用版本，但脚本职责要清晰，方便后续替换和扩展

换句话说：

> 可以先做轻量实现，但不能做会明显妨碍后续演进的临时烂尾结构。

### 3. `TestMap` 当前就是 Demo 主演示场景

现阶段不额外新建“大型教学关”或“完整大厅系统”。

`TestMap` 的职责明确为：

- 承担基础教学入口
- 承担基础战斗演示
- 承担最小完成闭环
- 作为后续继续扩展的主 Demo 场景

后续若确实需要分化教学场和演示场，也应在当前版本稳定后再决定。

## 二、任务分层

本阶段任务分为三类：

### P0：必须完成

完成后，项目应从“测试场”提升为“可试玩 Demo”。

### P1：强烈建议完成

完成后，项目会更像一个值得展示和传播的 Demo。

### P2：可选优化

完成后，观感更完整，但不应阻塞 P0 / P1。

## 三、P0 必做任务

## 任务 P0-1：建立 DemoController 纵切骨架

### 目标

为 `TestMap` 建立一个低耦合、可扩展的 Demo 控制层，统一承接：

- 教学流程
- UI 提示
- 完成判定
- 场景重开 / 返回

### 建议实现

- 继续以 `demo_scene_controller.gd` 作为流程入口
- 新增独立的 Demo HUD 脚本与场景节点
- 由控制器驱动 HUD，而不是由角色脚本直接更新 UI
- 明确脚本职责边界：
  - `character.gd` 负责角色输入与动作
  - 武器脚本负责武器行为与事件
  - Demo 控制器负责场景流程
  - Demo HUD 负责展示

### 完成标准

- `TestMap` 中存在清晰的 Demo 控制入口
- 流程与 UI 不再依赖角色脚本硬编码
- 后续可在不大改角色脚本的前提下继续扩展教学与目标流程

### 风险点

- 为了“通用性”过度设计成复杂任务系统
- 相反地，把所有流程继续堆回场景脚本或角色脚本

### 验证方式

- 打开 `TestMap`，确认 Demo 相关逻辑能集中从控制器理解
- 检查角色脚本没有被塞入大量教学 UI 逻辑

---

## 任务 P0-2：完成最小教学流程与 HUD

### 目标

让玩家进入场景后，能在无额外文档帮助下理解并完成基础操作。

### 推荐流程

1. 移动教学
2. 射击教学
3. 换弹教学
4. 指向训练目标
5. 完成提示

### 最低建议内容

- 屏幕上显示当前步骤提示
- 显示关键操作提示
- 显示返回菜单提示
- 完成后显示清晰完成状态

### 实现建议

- 使用简单状态机推进步骤
- 当前阶段优先“可读、可推进、可维护”
- 不要求立刻引入完整任务系统
- 提示文案要短，避免遮挡主视野

### 完成标准

- 玩家进入场景后能知道自己要做什么
- 至少存在移动、开火、换弹三步顺序推进
- 所有步骤完成后有明确完成提示

### 风险点

- 所有提示同时堆在屏幕上
- 判定条件过度依赖底层实现细节，导致后续难改
- 键位提示与 InputMap 实际定义不一致

### 验证方式

- 从主菜单进入 `TestMap`
- 不查看代码和文档，仅观察画面，判断是否能开始并完成一轮试玩

---

## 任务 P0-3：建立最小战斗验证目标

### 目标

让 Demo 不只是“会动会打”，而是能验证一条最小战斗闭环：

- 开火
- 命中
- 伤害生效
- 目标销毁
- 流程完成

### 建议内容

- 在 `TestMap` 中加入一个最小训练目标
- 目标需要具备：
  - 基础 HP
  - 被命中后可响应
  - HP 归零后销毁或失效
- Demo 流程最终要求玩家击毁该目标

### 设计约束

- 当前优先使用“训练目标”而不是完整 AI 敌人
- 先验证伤害链路，不急于做完整敌人行为树、巡逻、攻击等系统
- 结构上保留后续替换为正式敌人的空间

### 完成标准

- 玩家可通过武器命中该目标
- 目标受到伤害后有可见结果
- 目标销毁后流程能推进到完成状态

### 风险点

- 为了做训练目标，反而提前展开完整敌人系统
- 命中反馈、伤害判定、销毁逻辑混乱耦合

### 验证方式

- 正常完成教学后，击毁训练目标
- 确认完成提示稳定出现

---

## 任务 P0-4：补齐最小玩家状态闭环

### 目标

让玩家在 Demo 中不再只是“预先摆在场景里的一个可控角色”，而具备最小的状态完整性。

### 最低建议范围

- 角色出生或重置入口
- 基础生命值结构
- 受伤占位
- 死亡占位或失败占位

### 设计约束

- 当前只做 Demo 所需最低版本
- 不在这一阶段扩展完整成长、掉落、存档等体系
- 不强行完成全量动作状态树重构

### 完成标准

- 角色能被控制器重置，而不是完全依赖手摆场景
- 至少具备基础生存状态表达
- 后续加入敌人或危险源时，不需要重新推翻玩家状态入口

### 风险点

- 一上来把玩家状态系统做成全面重构
- 把 Demo 流程和角色底层状态机强耦合

### 验证方式

- 重开场景或重新开始时，玩家状态能正确恢复
- 玩家相关状态入口已具备后续扩展空间

---

## 任务 P0-5：补齐 Demo 收尾与返回逻辑

### 目标

让 Demo 有明确结束点，形成完整体验闭环。

### 建议内容

- 教学与目标完成后显示结果提示
- 提供：
  - 返回主菜单
  - 重新开始
- 保留现有 `ui_cancel` 返回逻辑，但避免与完成状态冲突

### 完成标准

- 玩家完成流程后，不会陷入“接下来该干嘛”的状态
- Demo 完成后有明确退出路径

### 风险点

- 完成状态没有锁定，提示反复变化
- 返回逻辑和普通试玩中的 `ui_cancel` 逻辑冲突

### 验证方式

- 完成全部步骤并击毁目标后，确认存在清晰收尾提示与返回路径

---

## 四、P1 强烈建议任务

## 任务 P1-1：增强射击反馈

### 目标

提升“开火瞬间”的手感与视觉反馈。

### 建议内容

- 强化枪口火光
- 强化射击动作或武器反馈
- 如成本合理，可加简单后坐感表现

### 完成标准

- 玩家开火时，能明确感受到动作已发生
- 视觉反馈不再显得过轻或过空

### 风险点

- 反馈过强影响读图
- 粒子、动画、脚本效果分散，后续难统一维护

---

## 任务 P1-2：加入命中反馈

### 目标

避免射击体验像“打空气”，提升互动感。

### 建议内容

- 训练目标命中特效
- 命中闪烁
- 受击反馈占位

### 完成标准

- 子弹命中对象时有明确可见反馈
- 玩家能判断自己的射击是否有效

### 风险点

- 过早把反馈逻辑和未来正式敌人系统绑死
- 命中表现与伤害判定不同步

---

## 任务 P1-3：优化镜头体验

### 目标

让角色操控与整体画面表现更平滑。

### 建议内容

- 基础平滑跟随
- 轻度视野偏移
- 如成本合理，可加入轻微屏幕震动

### 完成标准

- 镜头跟随不会明显生硬或割裂
- 不影响正常操作与读图

### 风险点

- 过度平滑导致拖沓感
- 震动频率过高影响体验

---

## 任务 P1-4：角色表现一致性检查

### 目标

让角色移动、朝向、持枪和射击表现更统一。

### 建议检查点

- 左右朝向是否稳定
- 武器方向是否与角色方向一致
- 站立、移动、开火切换是否自然
- 动画是否存在明显错帧或状态冲突

### 完成标准

- 基础动作表现没有明显出戏问题
- 玩家短时间试玩中不会频繁注意到动作异常

### 风险点

- 为修单点表现引入大量角色结构调整
- 把展示 polish 做成底层大重构

---

## 任务 P1-5：主菜单最小包装增强

### 目标

让项目启动画面更像一个 Demo，而非临时测试入口。

### 建议内容

- 显示项目标题
- 显示 Demo / Prototype 标识
- 保持开始按钮作为主操作入口
- 设置按钮继续占位，不强行展开设置系统

### 完成标准

- 主菜单具备基本展示识别度
- 玩家进入游戏后，能明确知道当前是项目 Demo

### 风险点

- 为做菜单包装引入过多无关 UI 工作
- 菜单视觉改动过大，打断当前开发节奏

## 六、第一步主要工作拆分

当前第一步不是“单独补 HUD”，而是完成第一条 Demo 纵切的骨架。

建议拆分为以下子任务：

### Step 1：建立 Demo 控制边界

- 检查并扩展 `demo_scene_controller.gd`
- 明确它负责流程状态、重开、返回、目标完成
- 避免继续把 Demo 流程分散到角色脚本中

### Step 2：加入 Demo HUD

- 在 `TestMap` 中加入 `CanvasLayer`
- 增加用于显示教学步骤与状态提示的 HUD 节点
- HUD 由控制器驱动，不直接读角色底层内部状态

### Step 3：定义最小流程状态机

- 至少包含：
  - `MOVE`
  - `FIRE`
  - `RELOAD`
  - `DESTROY_TARGET`
  - `COMPLETE`
- 每步只解决一个明确目标，避免流程提示堆叠

### Step 4：准备最小训练目标

- 放入一个可命中、可受伤、可销毁的训练目标
- 先不做复杂敌人 AI
- 但要保留以后替换为正式敌人的结构空间

### Step 5：接通完成判定与收尾提示

- 目标销毁后进入完成状态
- 显示完成提示
- 提供返回菜单或重新开始

这五步完成后，`TestMap` 就会从“测试地图”升级为“可演示的基础 Demo 场景”。

为避免返工，建议按以下顺序推进：

1. 建立 DemoController 纵切骨架
2. 完成最小教学流程与 HUD
3. 建立最小战斗验证目标
4. 补齐最小玩家状态闭环
5. 补齐 Demo 收尾与返回逻辑
6. 增强射击反馈
7. 增加命中反馈
8. 优化镜头体验
9. 检查角色表现一致性
10. 增强主菜单最小包装
11. 补充版本号与说明文本

## 八、验收清单

当以下条件都满足时，可认为“短期 Demo 下一阶段”基本完成：

- [ ] 启动游戏后稳定进入主菜单
- [ ] 玩家能从主菜单进入 Demo 场景
- [ ] 玩家进入场景后能理解基础操作
- [ ] Demo 至少包含移动、射击、换弹的引导流程
- [ ] Demo 中存在一个可击毁的训练目标
- [ ] 命中、伤害、生效、销毁这条最小战斗链路可被验证
- [ ] 流程完成后有明确完成提示
- [ ] 玩家可从完成状态返回主菜单或重新开始
- [ ] UI 至少能承担基础教学与状态提示
- [ ] 玩家状态入口已具备后续扩展空间
- [ ] 整体流程已可给外部朋友试玩，而不只是开发者自测

## 十、协作说明

后续如果在不同频道、不同 AI、不同开发阶段之间同步工作，建议统一遵守以下约束：

- 以本文件作为当前 Demo 推进目标基线
- 优先完成 P0，再决定是否插入 P1
- 若新增任务会明显扩大范围，应先标注为“后续阶段”而不是直接塞入当前 Demo 主线
- 如实际开发情况与本文件不符，应优先更新本文件，而不是仅在聊天中临时说明

这样可以减少跨频道协作时的目标漂移。
