# 短期 Demo 可行性方案

## 目标

在不急于进行完整结构升级的前提下，先做出一个**基本可玩、可展示、可吸引外部关注**的 Demo。

该阶段的重点不是把所有系统一次性做完整，而是优先建立一个足够顺畅的展示闭环，让试玩者和潜在协作者能够快速理解项目的潜力。

---

## 当前项目状态判断

### 已经具备的基础

- 已有可运行的主场景：`Scene/TestMap.tscn`
- 已有可控制的玩家角色原型：`Scene/Characters/character_host.tscn`
- 已有基础移动、重力、斜坡与动画切换逻辑：`Script/GamePlay/character.gd`
- 已有基础武器与投射物闭环：
  - `Scene/Weapons/ar_23_liberator.tscn`
  - `Scene/Weapons/5x50mm.tscn`
  - `Script/GamePlay/Weapons/Base/weapon_base.gd`
  - `Script/GamePlay/Weapons/Projectiles/projectile_base.gd`
- 已有部分 UI 骨架：`Scene/UI/main_menu.tscn`
- 已有数据与存档基础结构：`PlayerData / ItemDatabase / SaveManager`

### 当前主要问题

项目目前更像一个**可运行的玩法原型测试场**，而不是一个可对外展示的完整 Demo 流程。

主要不足在于：

- 缺少真正接管启动流程的大厅 / 主菜单逻辑
- 缺少教学场景
- 缺少清晰的场景切换链路
- 缺少教学引导与演示包装
- 角色操控虽有基础，但距离“展示级别的表演完成度”还有提升空间

---

## 核心判断

### 短期 Demo 是可行的

原因不是项目已经完整，而是它已经拥有了短期 Demo 最难的一部分：

- 角色原型
- 动画骨架
- 武器射击闭环
- 地图测试场景

因此，短期最优策略不是优先做大规模重构，而是：

> **用最少的结构改动，先把展示闭环拼出来。**

---

## 推荐的最小展示闭环

建议优先实现如下流程：

1. 启动游戏
2. 进入大厅 / 主菜单
3. 选择进入教学
4. 在教学场景中展示角色移动、射击、换弹与基础交互
5. 教学结束后可返回大厅或重新开始

只要这条链路成立，项目就会从“玩法测试工程”变成“可对外展示的 Demo”。

---

## 分阶段实施建议

## 第一阶段：补齐最小流程骨架

### 目标

让项目拥有完整、可进入、可退出、可展示的最小体验流程。

### 建议内容

1. **大厅场景**
   - 可以先复用 `Scene/UI/main_menu.tscn` 作为临时大厅入口
   - 最少提供以下选项：
     - 开始教学
     - 设置（可先占位）
     - 退出（可选）

2. **教学场景**
   - 可从 `Scene/TestMap.tscn` 复制出教学版本，降低初期工作量
   - 优先承载基础教学内容，而不是追求复杂关卡结构

3. **场景切换**
   - 建立大厅 → 教学 → 返回大厅的最小切换逻辑
   - 这是当前最缺失、但对 Demo 完整度提升最大的部分

### 阶段价值

完成后，项目将首次具备“对外可展示的结构感”。

---

## 第二阶段：补强角色展示效果

### 目标

让角色操控从“可运行”提升到“值得展示”。

### 建议优先项

1. **角色朝向与动作表现优化**
   - 左右移动时的朝向逻辑
   - 武器与身体表现的一致性

2. **开火反馈增强**
   - 保留并强化枪口火光
   - 补充开火动作表现、枪身反馈或更明确的射击演出

3. **镜头表现**
   - 适度平滑跟随
   - 轻微震动或反馈效果

4. **基础教学提示 UI**
   - 显示最基本的控制说明：
     - 移动
     - 开火
     - 换弹

### 阶段价值

完成后，角色会更接近“可以拿去吸引协作者”的演示水平。

---

## 第三阶段：补充展示友好度

### 目标

让 Demo 更适合发布到社区或展示给潜在协作者。

### 建议内容

- 开场标题或项目标识
- 一句简短项目说明
- Demo 版本号或阶段说明
- 结束页 / 返回入口
- 后续可加入协作招募信息（美术、音乐等）

### 阶段价值

提升 Demo 的完整性和传播友好度。

---

## 当前缺失内容清单（按优先级）

### 高优先级

- [ ] 大厅 / 主菜单真正接管启动流程
- [ ] 教学场景建立
- [ ] 场景切换逻辑
- [ ] 教学提示与最小 UI 指引
- [ ] 角色操控表现 polish

### 中优先级

- [ ] 命中反馈
- [ ] 枪械反馈增强
- [ ] 弹药与提示 UI
- [ ] 场景结束与返回逻辑

### 低优先级（短期可暂缓）

- [ ] 完整架构升级
- [ ] 联机能力落地
- [ ] 完整成长系统
- [ ] 深层数据结构扩展

---

## 实施策略建议

短期阶段建议遵循以下原则：

1. **优先把展示闭环跑通**，不要被大规模重构打断
2. **优先补最影响观感的功能**，尤其是角色操控、射击反馈、教学引导
3. **可在现有原型上迭代**，不必立刻迁完新旧架构
4. **先做一个能吸引关注的 Demo**，再进入更复杂的工程升级

---

## 结论

当前项目最适合采用的策略是：

> **先把原型包装成一个最小但完整的可玩 Demo，再进行更深层次的结构升级。**

这是当前阶段在开发成本、展示效果和后续协作吸引力之间，性价比最高的路线。
