# 武器设计

> 本文基于当前项目中的实际实现整理，重点说明 `WeaponBase` 的职责、数据结构、运行流程，以及当前这版脚本里已经落地的关键技术点。

---

## 1. 目标

- 为项目提供一个可复用的武器基类，支持不同枪械共享统一的开火、换弹和抛射物发射流程。
- 保持武器逻辑与角色逻辑解耦，角色只负责发起输入请求和承接 `bullet_fired` 信号。
- 为后续扩展半自动、全自动、霰弹、多弹丸散布、后坐力和精度系统、联机同步留出接口。

---

## 2. 当前实现位置

- 武器基类：`Script/GamePlay/Weapons/Base/weapon_base.gd`
- 投射物基类：`Script/GamePlay/Weapons/Projectiles/projectile_base.gd`
- 角色侧接入：`Script/GamePlay/character.gd`
- 玩家装备模块：`Script/GamePlay/Actors/Player/Modules/player_equipment_module.gd`

## 3. WeaponBase 的职责边界

`WeaponBase` 当前承担的是“武器内部状态机 + 发射请求生成器”的职责，而不是完整的投射物实例化器。

它负责：

- 管理弹匣、备用弹匣数量、换弹状态。
- 管理射速冷却。
- 管理当前散布值，并在持续射击后增加、停止射击后恢复。
- 根据 `pellets_per_shot` 和散布值生成一组射击方向。
- 通过 `bullet_fired(origin, dir, speed)` 信号把“生成子弹”的请求发给外部。
- 播放枪口闪光和武器动画。

它目前不负责：

- 直接实例化子弹节点。
- 命中判定。
- 伤害结算。
- 输入读取。
- 网络同步。

这种拆分的价值是：武器脚本只关心“是否允许开火”和“这次该发出哪些弹道参数”，而子弹创建、场景挂载和后续命中逻辑可以独立演进。

## 4. 导出参数设计

### 4.1 Ammo

- `mag_count`：剩余弹匣数量。
- `mag_size`：单个弹匣容量。
- `current_ammo`：当前弹匣内子弹数。

当前换弹模型是“整匣补满”：

- 开火时 `current_ammo -= 1`
- 换弹完成时 `mag_count -= 1`
- `current_ammo = mag_size`

这是一种非常适合原型阶段的实现，规则简单、易读，也方便未来替换成更复杂的补弹逻辑。

### 4.2 Fire

- `bullet_speed`
- `rounds_per_minute`
- `is_full_auto`
- `bullet_scene`
- `pellets_per_shot`

这里已经形成了一个比较清晰的抽象：

- `WeaponBase` 负责给出“这次射击的参数”
- 外部通过 `get_bullet_scene()` 取得子弹场景并实例化

`pellets_per_shot` 让同一套基类同时覆盖：

- 普通步枪：`1`
- 霰弹枪：`> 1`

### 4.3 Spread

- `min_spread_degrees`
- `max_spread_degrees`
- `spread_increase_per_shot_degrees`
- `spread_recovery_per_second_degrees`
- `current_spread_degrees`

这部分是当前脚本里最值得保留的技术点之一。它不是固定散布，而是一个“动态散布池”：

- 开火时增加
- 停火时逐渐恢复

因此它天然支持：

- 点射更准
- 连射更飘
- 不同武器差异化手感

### 4.4 Reload

- `reload_time`

换弹流程是异步的，使用：

- `await get_tree().create_timer(reload_time).timeout`

这样写的优点是实现直接、脚本状态清晰，不需要手动额外维护一个 reload timer 节点。

### 4.5 Muzzle Flash

- `flash_peak_energy`
- `flash_fade_time`

枪口闪光基于 `PointLight2D` 能量插值，而不是单纯开关可见性。当前做法是：

- 开火瞬间把 `energy` 拉到峰值
- 用 `Tween` 在极短时间内衰减到 0
- 结束后关闭 light

这是一个低成本但观感不错的视觉反馈方案。

## 5. 运行流程

### 5.1 角色侧输入

`character.gd` 当前负责：

- 解析 InputMap 中实际存在的开火和换弹动作名。
- 根据 `is_full_auto` 决定用 `is_action_pressed` 还是 `is_action_just_pressed`。
- 计算武器前向方向。
- 调用武器的 `try_fire()` 和 `start_reload()`。

也就是说，半自动与全自动的差异并不写死在角色状态机里，而是通过武器属性回传给角色输入层，这种耦合度是比较低的。

### 5.2 武器侧开火

`try_fire(dir)` 的顺序大致是：

1. 检查 `can_fire`
2. 检查 `is_reloading`
3. 检查 `current_ammo > 0`
4. 规范化方向，空方向时回退到 `Vector2.RIGHT`
5. 扣除弹药并发送 `ammo_changed`
6. 计算枪口位置
7. 根据散布生成一组抛射方向
8. 逐个发出 `bullet_fired`
9. 播放枪口闪光和射击动画
10. 增加当前散布
11. 锁定开火并启动射速计时器

这个流程已经具备较好的“事件先行”特征：武器不直接依赖某个具体子弹节点，从而避免把发射、挂载、命中处理耦合进同一个脚本。

### 5.3 角色侧生成投射物

`character.gd` 在 `_on_weapon_bullet_fired()` 中：

- 通过 `weapon.get_bullet_scene()` 取得子弹 PackedScene
- 实例化子弹
- 添加到当前战斗场景
- 设置位置、朝向、`direction`、`speed`

这说明当前的真实架构是：

`WeaponBase` 产生射击事件  
`Character` 承担项目中的临时 projectile factory

这对原型阶段是合理的，但后续如果武器数量和攻击类型增加，建议再抽离成独立的 projectile spawner 或 combat service。

### 5.4 投射物运行

`ProjectileBase` 当前职责非常纯粹：

- 按 `direction * speed` 直线移动
- 生命周期结束自动销毁
- 与 `body` 或 `area` 接触后，如果目标支持 `apply_damage()` 就造成伤害
- 发出 `hit` 信号后自毁

这让武器层和命中层之间的边界比较清楚。

## 6. 当前脚本中的关键技术点

### 6.1 信号驱动的发射解耦

`bullet_fired(origin, dir, speed)` 是目前这套设计的核心。

好处：

- 武器不需要知道子弹被挂到哪里。
- 同一个武器基类可以适配不同角色、敌人或测试场景。
- 后续如果要做联机，只需要把这个“发射事件”映射到同步层，而不是重写整套武器逻辑。

### 6.2 动态散布系统

`current_spread_degrees` 的引入，让精度从静态参数变成运行时状态。

它已经具备以下扩展潜力：

- 移动时附加散布惩罚
- 蹲伏和卧倒降低最小散布
- 不同武器拥有不同回复速度
- 瞄准状态缩小散布

### 6.3 RPM 转射击间隔

`_get_fire_interval()` 使用 `60.0 / rounds_per_minute` 计算单发间隔。

这比直接写 cooldown 秒数更符合枪械配置习惯，策划或调参时更直观。

### 6.4 多弹丸方向生成

`_build_projectile_directions()` 使用当前散布为每颗弹丸独立生成角度偏移。

这意味着霰弹枪并不是“1 次射击 + 1 条扩散扇形特效”，而是真正产出多条可独立计算碰撞的弹道。

### 6.5 Tween 枪口闪光

视觉反馈没有和动画系统强耦合，而是独立由 `_play_muzzle_flash()` 控制。

这样做的好处：

- 即使动画资源未完成，也能保留射击反馈。
- 后续可以单独替换成粒子、Sprite、Shader 或更复杂的灯光逻辑。

### 6.6 导出参数自校正

`_sanitize_exports()` 会在 `_ready()` 时校正常见非法值，例如：

- `pellets_per_shot >= 1`
- `max_spread_degrees >= min_spread_degrees`
- 各种 spread 增量不为负数

这能减少编辑器调参时把武器配坏的概率，是一个很实用的原型期防呆设计。

## 7. 当前版本的局限与后续建议

下面这些不是必须立刻修改的问题，但文档里需要明确，以免后续误判现状。

### 7.1 弹药模型较简化

当前 `mag_count` 更接近“剩余整弹匣数”，不是“背包总备弹数”。

这意味着：

- 换弹时无论当前弹匣剩多少，都消耗 1 个完整弹匣
- 暂时不支持更细的战术换弹、散装补弹、单发装填

如果未来要做更写实的武器系统，建议把：

- `reserve_ammo`
- `reload_mode`
- `ammo_per_reload_step`

拆出来单独建模。

### 7.2 空仓反馈尚未区分

`try_fire()` 在没子弹时直接返回，目前没有：

- 空仓音效
- 空击动画
- 自动触发换弹

这些都可以作为后续体验补强点。

### 7.3 换弹可中断规则尚未定义

`start_reload()` 当前一旦进入 `await`，没有中断路径。

后续需要明确：

- 翻滚是否打断换弹
- 切枪是否打断换弹
- 受击是否打断换弹

### 7.4 发射者与投射物工厂尚未抽离

现在是 `character.gd` 响应武器信号并实例化子弹。

这在原型期足够，但后续如果加入：

- 敌人武器
- 炮塔
- 联机同步
- 命中扫描武器

建议新增统一的 combat service 或 projectile spawner。

### 7.5 `current_spread_degrees` 既是导出值又是运行时值

目前它被 `@export` 暴露在编辑器中，同时在运行时持续变化。

这会带来一个语义问题：

- 它既像“默认配置”
- 又像“运行时状态”

后续更清晰的做法通常是：

- 导出 `initial_spread_degrees`
- 运行时维护非导出的 `current_spread_degrees`

不过在原型阶段，当前写法的优点是便于直接观察和调试。

## 8. 推荐的后续扩展方向

建议优先级从高到低如下：

1. 抽离 `ProjectileSpawner` 或 `CombatSpawnService`
2. 为 `WeaponBase` 增加空仓反馈信号，例如 `fire_blocked(reason)`
3. 把换弹规则扩展成可配置模式
4. 为散布系统叠加角色状态修正项
5. 为联机预留明确的 authority 归属与发射同步层

---

## 9. 当前设计结论

当前这版 `WeaponBase` 已经形成了一个不错的原型基线：

- 简洁
- 模块边界清楚
- 支持多种武器类型
- 有明确的后续演进空间

尤其值得保留的部分是：

- 信号驱动的发射解耦
- 动态散布系统
- 基于 RPM 的射速配置
- 多弹丸统一实现
- 低成本但有效的枪口闪光反馈

如果后续要继续迭代，最值得优先抽离的不是 `WeaponBase` 本身，而是角色里承担的子弹实例化职责。

## 10. 抬枪角与散布分离设计

为改善“持续射击只是在横向扩大散布扇区，缺少后坐力感”的问题，当前实现新增了一层独立于散布的“抬枪角累计”机制。

### 10.1 设计目标

- 保留原有散布系统，用于表现枪械稳定性下降后的随机偏差。
- 额外引入一个“射击基线被逐步抬高”的状态，用于表现连续射击时的后坐力趋势。
- 让玩家感受到两个不同层面的误差：
  - 散布：围绕当前基线的随机扩散
  - 抬枪角：整个基线逐步上扬

### 10.2 新增参数

- `initial_pitch_offset_degrees`
- `max_pitch_offset_degrees`
- `pitch_increase_per_shot_degrees`
- `pitch_recovery_per_second_degrees`
- `current_pitch_offset_degrees`

语义如下：

- `initial_pitch_offset_degrees`：武器的默认射击基线抬高角。
- `max_pitch_offset_degrees`：连续射击时允许累计到的最大抬高角。
- `pitch_increase_per_shot_degrees`：每次射击额外增加的抬枪角。
- `pitch_recovery_per_second_degrees`：停止射击后每秒恢复的抬枪角。
- `current_pitch_offset_degrees`：运行时当前实际抬枪角。

### 10.3 运行规则

- 武器初始化时，`current_pitch_offset_degrees = initial_pitch_offset_degrees`
- 每次射击后，抬枪角增加，但不会超过 `max_pitch_offset_degrees`
- 停止射击后，抬枪角逐步回落，但不会低于 `initial_pitch_offset_degrees`

这保证了两个边界：

- 不会超过最大抬枪角
- 不会恢复到低于初始射击基线

### 10.4 与散布的组合顺序

当前实现顺序为：

1. 先取得武器前向基线
2. 先应用抬枪角偏移
3. 再围绕偏移后的基线生成散布弹道

这种顺序更符合直觉：

- 后坐力先把枪口整体抬上去
- 然后散布再围绕“已经抬高后的枪口方向”产生随机误差

### 10.5 当前实现说明

抬枪角是按左右朝向分别处理的：

- 朝右射击时，角度向上偏转
- 朝左射击时，也会保持向上偏转，而不是错误地下压

这能避免简单固定正负旋转时，角色转身后“后坐力方向反了”的问题。

### 10.6 调参建议

如果想先得到比较明显但不夸张的手感，可以从下面的组合开始试：

- 轻武器：
  - `spread_increase_per_shot_degrees = 0.15 ~ 0.35`
  - `pitch_increase_per_shot_degrees = 0.2 ~ 0.5`
  - `pitch_recovery_per_second_degrees` 略高于单发增量乘以射速节奏
- 中型自动武器：
  - `spread_increase_per_shot_degrees = 0.25 ~ 0.6`
  - `pitch_increase_per_shot_degrees = 0.4 ~ 0.9`
  - `max_pitch_offset_degrees = 3 ~ 8`
- 重武器：
  - 适当提高 `pitch_increase_per_shot_degrees`
  - 适当降低 `pitch_recovery_per_second_degrees`
  - 让持续压枪成本更明显

经验上：

- 如果觉得“飘”，先减小散布增长
- 如果觉得“没有后坐力”，先增大单发抬枪角
- 如果觉得“停火后回正太慢”，提高抬枪恢复速度
- 如果觉得“连射太容易失控”，降低最大抬枪角或提高恢复速度
